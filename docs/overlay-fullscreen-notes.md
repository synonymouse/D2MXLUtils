## D2MXLUtils Overlay и полноэкранный режим Diablo II (MedianXL)

### 1. Что реализовано

- Оверлей-окно вынесено в отдельное окно Tauri с `label = "overlay"`.
- На Windows к окну применяются флаги:
  - `WS_EX_LAYERED` — поддержка прозрачности;
  - `WS_EX_TRANSPARENT` — клики проходят сквозь оверлей;
  - `WS_EX_TOOLWINDOW` — окно скрыто из Alt+Tab.
- Позиция и размер оверлея периодически синхронизируются с окном Diablo II по WinAPI (`FindWindowW`, `GetWindowRect`, `MoveWindow`, `SetWindowPos`).
- Оверлей стабильно работает в **оконном режиме** игры: рендерится поверх клиента, выглядит как классический overlay наподобие [`electron-overlay-window`](https://github.com/SnosMe/electron-overlay-window).

### 2. Поведение в полноэкранном режиме

#### На нативном Windows (работает)

При использовании D2GL (Glide/DDraw → OpenGL wrapper) в режиме "fullscreen":
- D2GL реализует fullscreen как **borderless windowed** (окно без рамок на весь экран), а не как эксклюзивный fullscreen.
- DWM (Desktop Window Manager) продолжает композитить окна.
- Наш overlay корректно отображается поверх игры.

**Проверено**: overlay работает в fullscreen на нативном Windows с D2GL.

#### В виртуализированных средах (Parallels и др.) — известное ограничение

Наблюдаемое поведение:
- В «полноэкранном» режиме (Alt+Enter в Diablo II / MedianXL через D2GL) оверлей **не виден**, пока игра активна на весь экран.
- Оверлей **появляется только при Alt+Tab** или когда окно игры теряет фокус.
- В оконном режиме overlay работает корректно.

Причина:
- Виртуальный GPU Parallels (и аналогичных гипервизоров) может "выносить" fullscreen-окно игры напрямую в Metal/хост-систему, минуя гостевой DWM.
- В результате layered-окна гостевой Windows физически не оказываются "выше" поверхности игры на хосте.
- Это поведение гипервизора, а не ОС или нашего кода.

### 3. Что мы пробовали для исправления (Parallels)

Все перечисленные подходы **не помогли** в виртуализированной среде:

| Подход | Описание | Результат |
|--------|----------|-----------|
| `HWND_TOPMOST` + `SetWindowPos` | Базовый top-most z-order | Не помогает |
| Owned window (`GWL_HWNDPARENT`) | Сделать overlay "owned" окном игры — owned-окна всегда поверх owner'а | Не помогает |
| `WS_EX_NOACTIVATE` | Запретить overlay получать фокус | Не помогает |
| `BringWindowToTop` | Принудительно поднять overlay наверх z-order | Ломает взаимодействие с игрой (мигающий курсор, невозможно кликать) |
| `IsIconic` / `IsWindowVisible` проверки | Дополнительная логика видимости | Не влияет на проблему |
| Отключение Fullscreen Optimizations | Галка в свойствах exe | Не помогает |
| Отключение Game Mode / Game Bar | Настройки Windows | Не помогает |

**Вывод**: проблема на уровне гипервизора (Parallels), который выводит 3D-контент игры напрямую в хост-систему, минуя гостевой оконный композитор.

### 4. Почему это не баг кода, а ограничение подхода

Текущая архитектура:

- Оверлей — это **отдельное WinAPI-окно**, живущее в нашем процессе Tauri.
- Мы **не вмешиваемся в рендер** Diablo II, не перехватываем DirectDraw/Direct3D/Glide/OpenGL и не рисуем внутри процесса игры.

В таком подходе:

- Работают режимы, где игра рендерится как обычное окно:
  - оконный режим;
  - borderless windowed (D2GL "fullscreen" на нативном Windows).
- В виртуализированных средах, где гипервизор "оптимизирует" fullscreen-окна, layered-окна поверх игры **могут не отображаться** — это поведение гипервизора, а не ошибка нашего кода.

### 5. Рекомендации для пользователей

#### Для пользователей на нативном Windows
- Overlay работает в fullscreen с D2GL.
- Никаких дополнительных действий не требуется.

#### Для пользователей Parallels / VMware / других виртуальных машин
- **Рекомендуется**: играть в оконном режиме (можно растянуть окно на весь экран без рамок через настройки D2GL).
- В этом режиме overlay работает корректно.

### 6. Альтернативные варианты (не реализованы)

1. **In-process overlay через хук `SwapBuffers`**
   - Инжект DLL в процесс игры, перехват OpenGL `SwapBuffers`, рисование UI прямо в backbuffer.
   - Работает в любом fullscreen, на любом окружении.
   - Требует значительных усилий: DLL, хуки, IPC, GL-рендер или ImGui-интеграция.

2. **Overlay на стороне macOS**
   - Отдельное нативное Swift/AppKit приложение, рисующее поверх окна Parallels на хосте.
   - Это уже совершенно другой проект.

3. **Модификация D2GL**
   - Добавить наш UI в ImGui-контекст D2GL.
   - Требует форка и поддержки отдельной версии D2GL.

### 7. Почему Steam / Discord Overlay «умеют» в fullscreen и чем они отличаются

- Steam Overlay, Discord Overlay и подобные решения **не используют отдельное layered-окно поверх игры**.
- Вместо этого они:
  - **инжектят DLL в процесс игры** (их код работает внутри того же процесса, что и рендер игры);
  - **перехватывают вызовы графических API** (DirectX 9/11/12, OpenGL, Vulkan, иногда старый DirectDraw) — hook'и на `Present` / `SwapBuffers` / DXGI swap chain и т.п.;
  - **рисуют свой UI напрямую в backbuffer / swap chain игры**, теми же средствами, что и сама игра.
- Универсальность достигается не «поддержкой каждой игры по отдельности», а **поддержкой нескольких графических API**:
  - пишется отдельная реализация хуков для D3D9, D3D11, D3D12, OpenGL, Vulkan;
  - дальше тысячи игр, использующих эти API, автоматически «подхватывают» overlay.
- Такой подход:
  - гораздо более мощный (overlay виден даже в эксклюзивном fullscreen);
  - но требует сложной инфраструктуры инжекта и хуков, глубокого понимания графических API;
  - потенциально конфликтует с античитами и повышает риск крашей.

В D2MXLUtils сознательно выбран более «мягкий» внешний подход (отдельное окно + WinAPI), чтобы:

- не лезть в рендер игры и не превращать проект в полноценный графический инжектор;
- минимизировать риски блокировок/античитов и упрощать сопровождение.

Поддержка настоящего fullscreen «как у Steam» потребовала бы отдельного, большого проекта по внедрению рендер-хука внутри клиента Diablo II / MedianXL.
