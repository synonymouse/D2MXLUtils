# Filter Preview — TODO

Backend уже отдаёт готовый решатель фильтра через Tauri-команду
`get_item_filter_action(config, item) -> FilterDecision` (`src-tauri/src/main.rs`).
Команда просто прогоняет `FilterConfig::decide()` на «воображаемом» `ItemDropEvent`
— без реального дропа в игре. Ждёт, когда вокруг неё построят UI.

## Идеи под эту команду

1. **Live-preview в редакторе DSL.** Справа от CodeMirror — панель с набором
   тест-предметов. Каждый раз при изменении текста правил фронт дёргает
   `get_item_filter_action` для каждого теста и подсвечивает
   hide / show / notify. Ошибки в логике видны сразу, без фарма.

2. **«Почему это скрыто/показано».** ПКМ по дропу в оверлее или в журнале —
   всплывающая панель: «матчнуло правило №N (строка 42), visibility=hide,
   notification=none». Полезно для отладки на сложных rule-set.

3. **Конструктор предмета.** Форма: выбрать class, quality, tier, ethereal,
   affixes → показать результат фильтра. Удобно тестировать редкие комбинации
   без фарма нужного предмета.

4. **Diff при смене профиля/правила.** Сохранить последние N реальных дропов
   из сессии (`seen_items` + `ItemDropEvent`), прогнать через новый конфиг,
   показать «после изменения станут скрыты: …, появятся: …».

## Почему не стоит просто портировать `decide()` на TypeScript

`FilterConfig::decide()` дергает `MatchContext`, регекспы из `base_name`,
tier-маппинг из `class_cache`. Дублирование этой логики на фронте гарантирует
расхождение с Rust-версией при любом изменении DSL. `get_item_filter_action`
— единственный источник правды.
