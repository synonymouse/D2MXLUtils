# Отчёт об отладке лутфильтра: Проблема кэширования

**Дата:** 2026-01-03
**Статус:** Найдена корневая причина, предложено решение

---

## Резюме проблемы

**Симптом:** Хук лутфильтра технически работает правильно, но предметы не скрываются согласно правилам фильтра.

**Корневая причина:** Игра **кэширует результаты** функции фильтрации. Хук вызывается только один раз при создании предмета (когда `iEarLevel=0`), а наш Scanner записывает значение позже (~200ms). Игра не пересчитывает видимость после изменения.

**Предложенное решение:** Использовать **битовую маску unit_id** в shared memory вместо записи в `iEarLevel`. Хук будет проверять список при каждом вызове.

---

## Хронология отладки

### 1. Исходная проблема (из предыдущих сессий)

- Хук установлен на `D2Sigma.dll.text+CBCD0` (сигнатура `83 EC 08 53 55 8B D9 56 57`)
- Трамплин генерируется и записывается корректно
- `iEarLevel` записывается успешно (verified OK)
- **НО предметы не скрываются**

### 2. Первая проблема: Инвертированная логика возврата

**Обнаружено через CE:**
- Вручную изменили код на `B0 00 C3` (mov al, 0; ret) → **ВСЕ предметы скрылись** ✓
- Изменили на `B0 01 C3` (mov al, 1; ret) → все предметы появились

**Вывод:** Логика возврата была инвертирована!

**Было (неправильно):**
```asm
Return_Hide:
    mov al, 1    ; AL=1 для hide
    ret

Return_Show:
    xor al, al   ; AL=0 для show
    ret
```

**Стало (правильно):**
```asm
Return_Hide:
    xor al, al   ; AL=0 для hide
    ret

Return_Show:
    mov al, 1    ; AL=1 для show
    ret
```

**Результат:** Логика исправлена, но предметы всё равно не скрываются.

---

### 3. Вторая проблема: Хук не вызывается

**Симптомы:**
- Счётчик вызовов хука = 0
- `g_last_unit_id` и `g_last_ear_level` остаются = 0
- В логах: `DEBUG: Hook counter BEFORE write: 0` и `AFTER write: 0`

**Проверка через CE:**
- JMP записан правильно: `E9 2B 33 A7 0C` → `jmp 17D00000` ✓
- Трамплин на месте по адресу `17D00000` ✓
- Breakpoint на `D2Sigma.dll.text+CBCD0` **срабатывает** ✓

**Вывод:** Функция вызывается, но счётчик не увеличивается → проблема с ASLR и логами от разных сессий.

---

### 4. Третья проблема: Timing и кэширование

**Ключевое наблюдение:**

После перезапуска с правильными адресами:
```
Filter action for 'Minor Healing Potion': show_item=false
  DEBUG: Hook counter BEFORE write: 0
set_item_visibility: pUnitData=0x17C91A80, addr=0x17C91AC8, value=2 (HIDE)
  -> Write verified OK: read back 2
  DEBUG: Hook counter AFTER write: 0  ← Хук НЕ вызывался между записью!
```

**Последовательность событий:**

1. **Игра создаёт предмет** (t=0ms)
   - Вызывает функцию фильтрации
   - `iEarLevel = 0` (ещё не записан)
   - Хук → Original_Code → игра решает показать
   - **Результат кэшируется** ← ПРОБЛЕМА!

2. **Scanner находит предмет** (t=200ms)
   - Записывает `iEarLevel = 2` (verified OK)
   - Но игра **больше не вызывает** функцию фильтрации!
   - Предмет остаётся видимым

3. **При установке breakpoint в CE:**
   - Игра останавливается
   - Что-то инвалидирует кэш (таймер? внутренний счётчик?)
   - При продолжении игра **пересчитывает видимость**
   - Вызывает функцию фильтрации снова
   - Теперь `iEarLevel = 2` → хук возвращает AL=0
   - **Предметы скрываются!** ✓

**Логи после breakpoint:**
```
DEBUG: Last hook check: unit_id=20, iEarLevel=2  ← Хук прочитал наше значение!
DEBUG: Last hook check: unit_id=21, iEarLevel=2
DEBUG: Last hook check: unit_id=22, iEarLevel=2
DEBUG: Hook counter changed: 1853 -> 1871  ← Функция работает активно!
```

**Вывод:** Хук работает **ПРАВИЛЬНО**, но игра кэширует результаты и не вызывает функцию повторно после изменения `iEarLevel`.

## Корневая причина

### Механизм кэширования в игре

Diablo II оптимизирует производительность кэшируя результаты функции фильтрации:

```
Игра создаёт предмет:
  ├─ Вызывает LootFilter_ShouldShowItem(pUnit)
  ├─ Результат: AL=1 (show) или AL=0 (hide)
  └─ Кэширует результат в структуре предмета

При отрисовке каждого кадра:
  └─ Использует КЭШИРОВАННОЕ значение
      (НЕ вызывает функцию заново!)
```

**Проблема нашего подхода:**
- Мы записываем `iEarLevel` **после** того как игра закэшировала результат
- Игра не знает что значение изменилось
- Предмет остаётся видимым

**Почему breakpoint помогает:**
- Пауза в выполнении → игра входит в особое состояние
- При продолжении → пересчитывает видимость всех предметов
- Вызывает функцию фильтрации заново → читает новый `iEarLevel` → скрывает

---

## Предложенное решение

### Подход: Shared Memory с битовой маской unit_id

Вместо записи в `iEarLevel` (которое читается **один раз** при создании предмета), будем использовать **список unit_id** в памяти, который хук проверяет **при каждом вызове**.

### Почему это работает?

```
Игра создаёт предмет (unit_id=123):
  ├─ Вызывает LootFilter_ShouldShowItem(pUnit)
  ├─ Хук проверяет: unit_id=123 в списке скрытых?
  ├─ НЕТ (Scanner ещё не добавил)
  └─ Возвращает AL=1 (show) → кэшируется

Scanner находит предмет:
  └─ Добавляет unit_id=123 в список скрытых

Игра вызывает функцию снова (при смене экрана/Alt/и т.д.):
  ├─ Хук проверяет: unit_id=123 в списке?
  ├─ ДА! (Scanner добавил)
  └─ Возвращает AL=0 (hide) → предмет скрывается!
```

**Преимущества:**
- ✅ Обходит кэширование (проверка актуальна при каждом вызове)
- ✅ Работает даже если хук вызывается ДО того как Scanner нашёл предмет
- ✅ Не зависит от внутренней логики игры
- ✅ Высокая производительность (O(1) с битовой маской)

## Детали реализации

### Вариант 1: Битовая маска (рекомендуется)

**Память:**
- Выделить **256 байт** = 2048 бит
- Каждый бит = 1 unit_id
- Поддержка до 2048 одновременных предметов (более чем достаточно)

**Алгоритм:**
```rust
// Scanner добавляет unit_id в маску:
fn hide_item(&self, unit_id: u32) {
    let bit_index = (unit_id % 2048) as usize;
    let byte_index = bit_index / 8;
    let bit_offset = bit_index % 8;

    let mut mask = vec![0u8; 256];
    self.ctx.process.read_buffer_into(self.g_hide_mask, &mut mask)?;

    mask[byte_index] |= 1 << bit_offset;

    self.ctx.process.write_buffer(self.g_hide_mask, &mask)?;
}
```

**Хук (asm):**
```asm
; ECX = pUnit
mov edx, [ecx+0Ch]           ; edx = unit_id

; Вычислить bit_index = unit_id % 2048
mov eax, edx
and eax, 0x7FF               ; eax = bit_index (unit_id % 2048)

; Вычислить byte_index = bit_index / 8
mov ebx, eax
shr ebx, 3                   ; ebx = byte_index

; Вычислить bit_offset = bit_index % 8
and eax, 7                   ; eax = bit_offset

; Создать маску: 1 << bit_offset
mov ecx, 1
shl ecx, al                  ; ecx = (1 << bit_offset)

; Прочитать байт из маски
movzx eax, byte ptr [g_hide_mask + ebx]

; Проверить бит
test eax, ecx
jnz Return_Hide              ; Если бит установлен -> скрыть

jmp Original_Code

Return_Hide:
    xor al, al               ; AL = 0 (hide)
    ret
```

**Производительность:**
- 1 деление (через AND - быстро)
- 2 чтения памяти
- 1 битовая операция
- **~5 наносекунд** - быстрее чем чтение iEarLevel!

---

### Вариант 2: Массив unit_id (проще, медленнее)

**Память:**
- 4 байта: count (количество элементов)
- 256 * 4 байта = 1024 байта: массив unit_id

**Алгоритм хука:**
```asm
mov edx, [ecx+0Ch]           ; edx = unit_id
mov eax, [g_hide_count]      ; количество скрытых
test eax, eax
jz Original_Code             ; список пуст

lea esi, [g_hide_list]       ; esi = начало массива

Loop_Start:
    cmp [esi], edx           ; сравнить с текущим unit_id
    je Return_Hide           ; если совпадает -> скрыть
    add esi, 4               ; следующий элемент
    dec eax
    jnz Loop_Start

jmp Original_Code
```

**Производительность:**
- O(n) где n = количество скрытых предметов
- Худший случай: ~50 предметов → ~50 наносекунд
- Всё ещё приемлемо, но хуже чем битовая маска

## Известные ограничения

### Коллизии unit_id
При использовании `unit_id % 2048`:
- Разные unit_id могут попасть на один бит
- False positive: предмет может скрыться хотя не должен

**Вероятность:**
- unit_id генерируется последовательно игрой
- На экране обычно <100 предметов
- Вероятность коллизии: ~100/2048 = 5%

**Решение при необходимости:**
- Увеличить размер маски до 512 байт (4096 бит) → вероятность <2.5%
- Или использовать полный массив (без модуля) → нет коллизий, но больше память

### Переполнение маски
При длительной игре unit_id растут → старые биты остаются установленными.

**Решение:**
- Очищать маску при входе в новую игру (`clear_cache()`)
- Или периодически очищать (каждые N секунд)

## Технические детали

### Исправленная логика возврата

```rust
// Return_Hide: (iEarLevel == 2 ИЛИ unit_id в маске)
let return_false_pos = code.len();
// xor al, al ; AL = 0 (HIDE - function returns 0 to hide)
code.push(0x30);
code.push(0xC0);
// ret
code.push(0xC3);

// Return_Show: (iEarLevel == 1)
let return_true_pos = code.len();
// mov al, 1 ; AL = 1 (SHOW - function returns 1 to show)
code.push(0xB0);
code.push(0x01);
// ret
code.push(0xC3);
```

**Проверено через CE:**
- `B0 00 C3` (mov al, 0; ret) → все предметы **скрываются** ✓
- `B0 01 C3` (mov al, 1; ret) → все предметы **показываются** ✓

### Адреса и ASLR

**Важно:** Из-за ASLR адреса меняются при каждом запуске:
- Логи показывают актуальные адреса текущей сессии
- В CE нужно использовать **относительные адреса** (D2Sigma.dll+offset)
- При отладке проверять что адреса в CE совпадают с логами
